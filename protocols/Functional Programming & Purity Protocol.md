# Functional Programming & Purity Protocol

**Version 1.0 — April 2026**
**Classification: Open Source — Ignis AI Labs**
**Enforcement Level: Mandatory for all production systems**

---

## Purpose

**Pure functions and isolated side effects are the foundation of systems that are predictable, testable, debuggable, and safe for AI agents to modify.** When every function produces the same output for the same input and side effects are confined to explicit boundaries, the entire codebase becomes a composable, verifiable machine — not a fragile web of hidden dependencies.

This protocol exists because Ignis AI Labs builds systems where correctness is non-negotiable: **high-performance trading bots** that must execute deterministically under load, **AI inference pipelines** (SAGE) that must produce reproducible results, **blockchain infrastructure** (Unavis) where immutability is law, and **AI agent orchestration systems** (Shadow Clone) where multiple autonomous agents must modify code without corrupting shared state. Functional purity is not a style preference — it is an engineering requirement.

**Core axioms driving this protocol:**

- **Predictability over convenience.** A pure function can be tested, memoized, parallelized, and formally verified. An impure function with hidden dependencies cannot.
- **Immutability over mutation.** Values that never change eliminate entire categories of concurrency bugs, stale-state errors, and debugging nightmares.
- **Explicit over implicit.** Side effects, dependencies, and error conditions must be visible in function signatures — never hidden in implementation details.
- **Composition over complexity.** Small, pure functions compose into complex behavior. Monolithic impure functions resist modification and testing.
- **Types as contracts.** The type system should make illegal states unrepresentable, catching bugs at compile time rather than in production.

This protocol sits alongside the Security Audit Protocol, Code Quality Protocol, and Code Efficiency Protocol as a mandatory compliance standard. It applies to all languages used at Ignis AI Labs (TypeScript, Rust, Python, Go, Solidity) and to all code — whether written by humans or generated by AI agents.

## How to use this protocol

This protocol is a **verification checklist**, not a guideline document. Every item is designed to be objectively assessable by an AI coding agent or human reviewer.

**Status markers:**
- `[ ]` — Not yet verified
- `[x]` — Passing
- `[!]` — Failing (requires remediation)
- `[~]` — Partial compliance (document gaps)
- `[N/A]` — Not applicable to this codebase

**Severity tags:**
- **[CRITICAL]** — Deployment blocker. Must pass before merge. Violations risk data corruption, race conditions, financial loss, or security compromise.
- **[HIGH]** — Must fix within current sprint. Violations significantly degrade reliability, testability, or maintainability.
- **[MEDIUM]** — Fix within two sprints. Violations affect code quality but do not pose immediate operational risk.
- **[LOW]** — Address when modifying the file. Style preferences and minor optimizations.

**Enforcement tiers:**
- **Automated**: Enforced by linters, compilers, or CI/CD gates. No human judgment required.
- **AI-Verified**: Verified by AI coding agents during code review. Requires structured reasoning.
- **Human-Reviewed**: Requires human architectural judgment. Applied during design reviews and PR approvals.

**For AI coding agents (Claude Code, Shadow Clone):** Treat every `[CRITICAL]` item as a hard constraint. Before generating or modifying code, verify that your output does not introduce violations. When uncertain, flag for human review. Use the measurable criteria in each section to self-assess compliance.

**For human reviewers:** Use this as a structured review checklist. Start with `[CRITICAL]` items, then `[HIGH]`. Use the waiver process for justified exceptions — document the rationale and set a remediation timeline.

**Exception process:** Any `[CRITICAL]` or `[HIGH]` rule may be waived with written justification approved by a Tech Lead or Principal Engineer. Waivers must specify: the rule being waived, the business justification, the risk mitigation in place, and the remediation deadline. All active waivers are tracked in the project's compliance dashboard.

## 1 — Pure function enforcement

Pure functions are the atomic unit of reliable software. A function is pure if and only if it satisfies two conditions: **determinism** (same inputs always produce the same outputs) and **no observable side effects** (no interaction with the external world). Referential transparency — the ability to replace a function call with its result without changing program behavior — is the practical test of purity. This property enables memoization, safe parallelization, equational reasoning, and trivial testability.

**Target: ≥80% of business logic functions must be pure.** Gary Bernhardt's heuristic allocates 90% functional core, 5% imperative shell, and 5% transient imperative glue. Mark Seemann's research confirms that "a surprisingly large proportion of problems can be rephrased and organised in such a way that" the impure-pure-impure sandwich applies.

### Purity verification checklist

- [ ] **[CRITICAL]** Every function in the domain/business logic layer is pure — takes all inputs as explicit parameters, returns a value, and produces no observable side effects
- [ ] **[CRITICAL]** No pure-labeled function reads the system clock (`Date.now()`, `SystemTime::now()`, `time.time()`, `time.Now()`, `block.timestamp`)
- [ ] **[CRITICAL]** No pure-labeled function uses randomness (`Math.random()`, `rand::thread_rng()`, `random.random()`, `crypto/rand`)
- [ ] **[CRITICAL]** No pure-labeled function performs I/O — no file system access, network calls, database queries, or `console.log`/`println!`/`print()`/`fmt.Println`
- [ ] **[CRITICAL]** No pure-labeled function reads or writes global/module-level mutable state, environment variables (`process.env`, `std::env`, `os.environ`, `os.Getenv`), or singleton registries
- [ ] **[CRITICAL]** No pure-labeled function mutates its input arguments — all data flows out via return values only
- [ ] **[HIGH]** Pure functions do not throw exceptions for control flow — expected failures are returned as `Result`/`Either` types (see Section 6)
- [ ] **[HIGH]** Every function is clearly categorized as pure or effectful — effectful functions are named with verb prefixes (`fetch`, `save`, `send`, `log`, `emit`) and documented as side-effectful
- [ ] **[HIGH]** Time, randomness, and configuration are injected as explicit parameters: `isExpired(expiryDate, now)` not `isExpired(expiryDate)` with internal clock reads
- [ ] **[MEDIUM]** Pure functions are co-located in dedicated modules/files separate from effectful code (e.g., `domain/`, `core/`, `logic/` directories vs `adapters/`, `infrastructure/`, `shell/`)
- [ ] **[MEDIUM]** Referential transparency is verified by confirming that memoizing the function would not change program behavior
- [ ] **[LOW]** Idempotent operations in distributed systems use idempotency keys or UPSERT semantics to achieve effectively-exactly-once delivery

**Measurable criteria:**

| Metric | Threshold | Enforcement |
|---|---|---|
| Pure function ratio in business logic | ≥80% | AI-Verified |
| Side-effectful functions with verb-prefix naming | 100% | Automated (naming lint) |
| Functions reading global state in core modules | 0 | Automated (lint rules) |
| Functions with injected time/random dependencies | 100% of functions using time/random | Human-Reviewed |

## 2 — Immutability standards

**Immutable data structures eliminate entire categories of bugs** — race conditions, stale state, unintended aliasing, and debugging sessions that chase phantom mutations through call stacks. Rich Hickey's core insight is that a value is immutable by nature — the number 5 does not change, and your data should not either. Identity (a logical entity) should be a controlled sequence of immutable values over time, not a mutable object that changes in place.

Modern persistent data structures (Hash Array Mapped Tries) achieve immutable updates in **O(log₃₂ n)** time — effectively O(1) for practical sizes — by sharing structure between versions. Clojure's implementation stores millions of elements in 4–6 tree levels. The performance cost of immutability is typically **2–3× slower** for sequential single-threaded mutations, but often **faster in concurrent scenarios** due to eliminated locking.

### Immutability checklist

- [ ] **[CRITICAL]** All variable bindings use immutable declarations by default — `const` (JS/TS), `let` without `mut` (Rust), `Final` (Python), `const` for compile-time values (Go/Solidity)
- [ ] **[CRITICAL]** No direct mutation of objects, arrays, maps, or sets passed as function parameters — return new copies or use structural sharing (Immer `produce()`, spread with care, Rust `clone()`)
- [ ] **[CRITICAL]** No shared mutable state between concurrent execution contexts (threads, goroutines, async tasks, actors) — use immutable message passing, channels, or `Arc<Mutex<T>>` with explicit locking
- [ ] **[HIGH]** All data types in domain models use immutable representations — `Readonly<T>` (TS), `frozen=True` dataclasses (Python), immutable structs (Rust), `immutable`/`constant` (Solidity)
- [ ] **[HIGH]** Deep immutability is enforced for nested objects — `DeepReadonly<T>` (TS), `Object.freeze()` for runtime enforcement, recursive `frozen=True` (Python)
- [ ] **[HIGH]** Collection types use immutable variants — `ReadonlyArray<T>`, `ReadonlyMap<K,V>`, `ReadonlySet<T>` (TS), `tuple`/`frozenset` (Python), immutable borrows `&[T]` (Rust)
- [ ] **[HIGH]** Configuration objects are frozen at initialization and never modified — use `as const` (TS), `const` blocks (Rust), `Final` annotations (Python)
- [ ] **[MEDIUM]** State updates in UI/application layers use immutable update patterns — Immer `produce()` (React/Redux), Elm-style reducers, or persistent data structures
- [ ] **[MEDIUM]** Value objects in domain models implement structural equality — two instances with identical data are considered equal regardless of reference identity
- [ ] **[LOW]** Copy-on-write (`Cow<T>` in Rust, Immer proxies in JS) is used where performance requires deferred cloning
- [ ] **[LOW]** Builder patterns freeze the constructed object upon completion — no mutation after `.build()`

### When mutation is acceptable

Mutation is permitted under the following conditions, each of which must be documented:

- **Local mutation within a pure function boundary.** If mutation is invisible to the caller (encapsulated within the function body, not escaping via return value or closure), the function remains pure from the outside. Haskell formalizes this with the `ST` monad; Rust's ownership system guarantees local mutation stays local.
- **Performance hot paths with benchmarked evidence.** When profiling demonstrates that immutable operations are a bottleneck (e.g., tight inner loops processing millions of elements), local `let mut` (Rust) or mutable accumulators are acceptable. Clojure's transients provide a model: create a transient, mutate locally, convert back to persistent — **2–3× faster** than direct persistent construction.
- **FFI boundaries with external mutable APIs.** When interfacing with C libraries, hardware drivers, or legacy systems that require mutable pointers.

Every exception must include a code comment: `// MUTATION: [justification] — [ticket/waiver reference]`.

## 3 — Side effect isolation

The **"functional core, imperative shell"** architecture is the central organizing principle of this protocol. All decision logic — the rules, transformations, and computations that define business behavior — lives in a pure functional core. All interactions with the outside world — databases, networks, file systems, clocks, random number generators — live in a thin imperative shell that serves as an adapter between the pure core and the external world. Gary Bernhardt's research demonstrates that this separation eliminates the need for most mocking, because the core is testable with simple inputs and outputs, and the shell is testable with integration tests against real dependencies.

Mark Seemann refined this into the **"impureim sandwich"** (also called "REad-CAlculate-WRite"): read impure data, calculate with pure functions, write impure results. Once writing begins, do not go back to reading — this prevents the sandwich from collapsing into interleaved impure/pure chaos.

### Side effect isolation checklist

- [ ] **[CRITICAL]** Architecture follows the functional core / imperative shell pattern — business logic is pure, I/O happens only in adapters and entry points
- [ ] **[CRITICAL]** No database queries, API calls, file reads, or network requests inside domain/business logic functions — all external data is passed in as parameters
- [ ] **[CRITICAL]** No global state mutation from within the functional core — state changes flow outward via return values
- [ ] **[HIGH]** Side effects follow the impureim sandwich: Read (impure) → Calculate (pure) → Write (impure) — once writing begins, no reading occurs within the same operation
- [ ] **[HIGH]** Command-Query Separation (CQS) is applied: functions either return data (queries, no side effects) or perform actions (commands, may have side effects), never both
- [ ] **[HIGH]** All external dependencies are injected, not imported directly within business logic — database repositories, HTTP clients, message queues, loggers, clocks are provided via dependency injection, function parameters, or the Reader pattern
- [ ] **[HIGH]** Logging is treated as a side effect — pure functions return log-worthy data as part of their return value; the shell handles actual log emission
- [ ] **[HIGH]** Side effect boundaries align with hexagonal architecture ports — inbound ports (HTTP handlers, CLI entry points, message consumers) and outbound ports (repositories, API clients, event publishers)
- [ ] **[MEDIUM]** Effectful functions are organized in dedicated adapter/infrastructure modules, clearly separated from domain modules in the directory structure
- [ ] **[MEDIUM]** Effect systems (Effect-TS, ZIO, Cats Effect) or effect markers (`IO` type, `@impure` decorator) are used in projects where the language supports them, making side effects visible in type signatures
- [ ] **[MEDIUM]** Repository pattern is used for all database access — repositories implement interfaces defined by the domain, not the other way around (dependency inversion)
- [ ] **[LOW]** CQRS (Command Query Responsibility Segregation) is evaluated for modules with complex read and write patterns — separate read models from write models when query and command complexity diverge significantly

**Measurable criteria:**

| Metric | Threshold | Enforcement |
|---|---|---|
| Side-effectful code in domain/core modules | 0% | Automated (import restrictions, lint rules) |
| External dependencies injected vs hard-coded | 100% injected | AI-Verified |
| Functions violating CQS (return value + side effect) | 0 in business logic | Human-Reviewed |
| Integration test coverage of shell/adapter code | ≥80% | Automated (coverage tools) |

## 4 — Composition patterns

Function composition is how pure functions scale to complex behavior. Instead of monolithic functions with interleaved logic, small focused functions compose via **pipe** (left-to-right data flow), **flow** (creating reusable composed functions), or **method chaining** (iterator chains in Rust, fluent APIs). The key insight: **functions of arity 1 compose most naturally** — currying and partial application convert multi-argument functions into composable unary functions.

### Composition checklist

- [ ] **[HIGH]** Pipelines and chains are used for sequential data transformations instead of deeply nested function calls — `pipe(data, fn1, fn2, fn3)` over `fn3(fn2(fn1(data)))`
- [ ] **[HIGH]** Data-last argument ordering is used for functions intended for composition — the data parameter is the final argument, enabling partial application of configuration parameters
- [ ] **[HIGH]** Monadic composition (`andThen`/`chain`/`flatMap`) is used for chaining operations that return `Result`/`Either`/`Option` — never unwrap intermediate results to re-wrap them
- [ ] **[HIGH]** Higher-order functions (`map`, `filter`, `reduce`, `flatMap`) are used instead of manual loops for data transformations (see Section 8 for exceptions)
- [ ] **[MEDIUM]** Pipelines are limited to **≤7 steps** — beyond this, extract named sub-pipelines with descriptive names that document the transformation stage
- [ ] **[MEDIUM]** Currying and partial application are used only when they produce meaningful specialized functions (e.g., `addTax = multiply(1.08)`) — not as default style for all functions
- [ ] **[MEDIUM]** Composition over inheritance — behavior reuse is achieved through function composition, higher-order functions, and mixins, not class hierarchies
- [ ] **[LOW]** Point-free (tacit) style is used only when function names are self-documenting — `arr.map(toUpperCase)` is clear, `arr.map(R.compose(R.join(','), R.map(R.prop('name'))))` is not
- [ ] **[LOW]** Third-party functions used in point-free style are wrapped at boundaries to prevent arity mismatch bugs (the `['1','2','3'].map(parseInt)` class of errors)

**Language-specific composition patterns:**

| Language | Primary composition tool | Pipeline operator |
|---|---|---|
| TypeScript (fp-ts) | `pipe(value, fn1, fn2)`, `flow(fn1, fn2)` | N/A (use `pipe`) |
| TypeScript (Effect) | `Effect.gen(function*() { })`, `pipe()` | N/A |
| Rust | `.iter().map().filter().collect()` | Method chaining |
| Python | `functools.reduce()`, generator pipelines | N/A |
| Go | First-class functions, functional options | N/A |

## 5 — State management discipline

Uncontrolled state is the root cause of most production bugs in complex systems. This protocol mandates that state transitions are **explicit, typed, and testable** — never implicit mutations scattered across the codebase. The reducer pattern (`(state, action) → newState`) treats state updates as pure functions. State machines encode valid transitions at the type level, making impossible states impossible. Event sourcing preserves the complete history of state changes as an immutable log, enabling audit trails and temporal debugging.

### State management checklist

- [ ] **[CRITICAL]** No global mutable state in the application — no mutable singletons, no module-level `let` variables modified at runtime, no mutable static variables
- [ ] **[CRITICAL]** In concurrent systems, state is either isolated per actor/thread (actor model, goroutine-local) or protected by explicit synchronization primitives (`Mutex`, `RwLock`, channels) — never accessed via unprotected shared mutable references
- [ ] **[CRITICAL]** State transitions in financial, trading, or safety-critical paths use typed state machines — valid transitions are enforced at the type level (Rust enums, TypeScript discriminated unions, XState)
- [ ] **[HIGH]** Entities with **≥3 distinct states** are modeled as explicit state machines with typed transitions — not boolean flags or string status fields
- [ ] **[HIGH]** State updates follow the reducer pattern: `(currentState, action) → newState` as a pure function — no direct state mutation outside the reducer
- [ ] **[HIGH]** Dependency injection replaces all global state access — services, configuration, and repositories are injected via constructors, function parameters, or the Reader/Context pattern
- [ ] **[HIGH]** State machines encode illegal transitions as compile-time errors — adding a new state variant forces handling in all transition functions (exhaustive matching)
- [ ] **[MEDIUM]** Event sourcing is used for domains requiring audit trails, temporal queries, or state replay — events capture business intent (`OrderPlaced`, `PaymentReceived`), not data diffs
- [ ] **[MEDIUM]** The actor model or channel-based message passing is preferred over shared-memory concurrency for inter-component communication
- [ ] **[MEDIUM]** Elm Architecture (Model-Update-View) or equivalent unidirectional data flow is used for UI state management — state flows down, events flow up
- [ ] **[LOW]** CQRS is applied when read and write models have significantly different performance or structural requirements

**Anti-patterns that constitute automatic failures:**

| Anti-pattern | Severity | Detection |
|---|---|---|
| Singleton mutable state accessed from business logic | CRITICAL | Automated (lint for global `let`/`var` modification) |
| Shared mutable state across threads without synchronization | CRITICAL | Automated (Rust: compiler; Others: race detector, lint) |
| Boolean flags encoding state (e.g., `isActive && !isDeleted && isPending`) | HIGH | AI-Verified (flag for state-like boolean combinations) |
| Module-level variables modified after initialization | HIGH | Automated (lint for module-scope mutation) |
| Implicit state in closures that capture mutable outer variables | MEDIUM | AI-Verified |

## 6 — Error handling as data

**Errors are values, not exceptions.** This protocol mandates typed error handling using `Result<T, E>` / `Either<E, A>` for all expected failures and `Option<T>` / `Maybe<A>` for all values that may be absent. Exceptions are reserved for truly unrecoverable conditions — out-of-memory, stack overflow, and programming errors that indicate bugs rather than expected failure modes.

Scott Wlaschin's railway-oriented programming provides the mental model: data flows along two tracks (success and failure), operations compose via `andThen`/`chain`/`flatMap`, and once on the failure track, subsequent operations are bypassed until an explicit recovery point. This eliminates hidden control flow, untyped error strings, and swallowed exceptions.

**Three error categories determine the handling strategy:**

- **Domain errors** (invalid input, business rule violations, insufficient balance) → `Result`/`Either` — part of the domain model, handled explicitly
- **Infrastructure errors** (network timeout, database unavailable, file not found) → `Result`/`Either` if recovery is possible; exception/panic if not
- **Panics** (out-of-memory, stack overflow, assertion failures) → Let the runtime handle; catch only at system boundaries for graceful shutdown

### Error handling checklist

- [ ] **[CRITICAL]** All expected failure conditions in domain logic use `Result<T, E>` / `Either<E, A>` — never thrown exceptions
- [ ] **[CRITICAL]** All values that may be absent use `Option<T>` / `Maybe<A>` — never `null`, `undefined`, or `nil` in typed code paths
- [ ] **[CRITICAL]** No swallowed errors — every error is handled, logged, propagated, or explicitly converted. Empty `catch {}` blocks, ignored `Result` values, and unchecked `.unwrap()` in production code are deployment blockers
- [ ] **[CRITICAL]** Error types are defined as typed enums/discriminated unions — never raw strings, generic `Error` objects, or unstructured data
- [ ] **[HIGH]** Railway-oriented programming (monadic chaining) is used for multi-step operations that can fail — `pipe(input, validate, E.chain(process), E.chain(save), E.fold(handleError, handleSuccess))`
- [ ] **[HIGH]** The `?` operator (Rust) or `andThen`/`chain` (TS/others) is used for error propagation — never manual `if err != nil { return err }` chains that lose context
- [ ] **[HIGH]** Error context is preserved during propagation — `fmt.Errorf("operation: %w", err)` (Go), `.context("operation")` (Rust/anyhow), `mapErr(addContext)` (TS)
- [ ] **[HIGH]** Exception boundaries exist at all system edges — HTTP handlers, message consumers, CLI entry points, and `main()` catch and convert exceptions to structured error responses
- [ ] **[HIGH]** Third-party code that throws exceptions is wrapped at the boundary — `Result.fromThrowable()` (neverthrow), `try/except` to `Result` (Python), `catch_unwind` (Rust) — converting to `Result` immediately
- [ ] **[MEDIUM]** Error accumulation (collecting all errors instead of failing fast) is used for validation — form input, batch processing, configuration parsing — using `Validated` (Cats), `Validation` (ZIO), or manual accumulation
- [ ] **[MEDIUM]** Each module defines ≤10 error variants — if exceeded, the module should be split
- [ ] **[LOW]** Error types implement `From` trait (Rust) or equivalent conversion for automatic error type adaptation across module boundaries

**Language-specific error handling tools:**

| Language | Result type | Option type | Error propagation | Recommended library |
|---|---|---|---|---|
| Rust | `Result<T, E>` | `Option<T>` | `?` operator | thiserror (libs), anyhow (apps) |
| TypeScript | `Result<T, E>` | `Option<A>` | `.andThen()` / `pipe` + `chain` | neverthrow or Effect-TS |
| Python | `Result[T, E]` | `Maybe[T]` | `flow()` + `bind()` | dry-python/returns |
| Go | `(T, error)` | `*T` (nil check) | `if err != nil` + `%w` wrapping | stdlib errors package |
| Solidity | Custom errors | N/A (revert) | `revert CustomError()` | Solidity 0.8.4+ custom errors |

## 7 — Type-driven design

**The type system is the first line of defense.** Making illegal states unrepresentable through types catches bugs at compile time — before code runs, before tests execute, before deployment. Yaron Minsky (Jane Street) and Scott Wlaschin established the principle: if the compiler rejects invalid states, they cannot occur at runtime. Alexis King's "parse, don't validate" refines this into a practical pattern: transform unstructured input into typed domain objects at system boundaries, then operate on proven-valid data throughout the system.

### Type-driven design checklist

- [ ] **[CRITICAL]** All external input is parsed into domain types at system boundaries — API request bodies, CLI arguments, configuration files, environment variables, and database results are validated and transformed into typed structures before entering the domain layer. Use Zod (TS), Pydantic (Python), serde (Rust), or equivalent.
- [ ] **[CRITICAL]** All discriminated unions / tagged unions are handled with exhaustive pattern matching — no unguarded default/wildcard cases that hide unhandled variants. Use `never` type exhaustiveness checks (TS), `match` without wildcard (Rust), `match`/`case` (Python 3.10+).
- [ ] **[HIGH]** Domain primitives are wrapped in branded types / newtypes — `UserId`, `EmailAddress`, `USD`, `Meters` instead of raw `string`, `number`, `f64`. This prevents argument transposition (`transfer(from, to, amount)` with raw types is dangerous; `transfer(fromAccount: AccountId, toAccount: AccountId, amount: USD)` catches swaps at compile time).
- [ ] **[HIGH]** Smart constructors enforce invariants at construction time — `NonEmptyList.fromArray()` returns `Option<NonEmptyList<T>>`, `PositiveInteger.parse()` returns `Result<PositiveInteger, ValidationError>`. Raw constructors for constrained types are private.
- [ ] **[HIGH]** Adding a new variant to any union/enum type triggers compile errors in all code that handles it — verified by exhaustive matching without wildcards
- [ ] **[HIGH]** Phantom types are used for state-dependent operations where the state transition must be verified at compile time — `Connection<Open>` vs `Connection<Closed>`, `FormData<Validated>` vs `FormData<Unvalidated>`
- [ ] **[MEDIUM]** No validation functions that return `bool` for domain parsing — all return typed results: `parseEmail(input: string): Result<EmailAddress, ValidationError>` not `isValidEmail(input: string): boolean`
- [ ] **[MEDIUM]** Functions are total (defined for all inputs of their stated type) — if a function cannot handle some inputs, narrow the input type rather than throwing exceptions or returning undefined behavior
- [ ] **[LOW]** Domain types encode units of measurement to prevent unit-mixing bugs (the Mars Climate Orbiter class of errors) — `Distance<Miles>` vs `Distance<Kilometers>` via phantom types or newtypes

**Measurable criteria:**

| Metric | Threshold | Enforcement |
|---|---|---|
| Domain primitives wrapped in branded/newtypes | ≥90% of domain concepts | AI-Verified |
| External input parsed at boundaries | 100% | Human-Reviewed |
| Discriminated unions exhaustively matched | 100% (no unguarded wildcards) | Automated (compiler + lint) |
| Validation functions returning typed results (not bool) | 100% for domain parsing | AI-Verified |

## 8 — Declarative over imperative

Declarative data transformations state **what** is being computed, not **how** to compute it. When a reviewer sees `users.filter(isActive).map(toDTO)`, the intent is immediately clear. When they see a `for` loop, they must read the body to determine whether it is a map, filter, find, reduce, or something more complex. Declarative code communicates intent; imperative code communicates mechanism.

Lazy evaluation eliminates the performance cost of declarative chaining. **Rust iterators compile to identical machine code as hand-written loops** — zero-cost abstractions verified by benchmarks. Python generators process elements one at a time with O(1) memory. Only in eager languages (JavaScript, Python lists) does chaining `.filter().map()` create intermediate allocations, and even then the overhead is negligible for collections under 10,000 elements.

### Declarative patterns checklist

- [ ] **[HIGH]** Simple data transformations (mapping, filtering, accumulation) use `map`, `filter`, `reduce`, `flatMap` — not manual `for`/`while` loops
- [ ] **[HIGH]** `forEach` (JS/TS) is used only for genuine side effects (logging, DOM updates) — never for building arrays or accumulating results. Use `map` or `filter` instead
- [ ] **[HIGH]** No mutable accumulator pattern (`let arr = []; items.forEach(i => { arr.push(transform(i)) })`) — use `items.map(transform)` or `items.filter(pred).map(transform)`
- [ ] **[HIGH]** No `reduce` with spread operator anti-pattern (`(acc, x) => ({...acc, [x.key]: x.val})`) — this is O(n²). Use `Object.fromEntries()`, mutable accumulator within reduce, or `Map`
- [ ] **[MEDIUM]** Maximum loop nesting depth is **≤2** — deeper nesting must be refactored into named helper functions
- [ ] **[MEDIUM]** Hot paths (>100k elements) use lazy evaluation to avoid intermediate allocations — Rust iterators, Python generators, Java streams, or single-pass reduction
- [ ] **[MEDIUM]** Optional chaining (`x?.foo?.bar`) or `Option.map()` chains replace imperative null-checking cascades
- [ ] **[LOW]** Imperative code is explicitly acceptable in: complex stateful algorithms (graph traversal, dynamic programming), performance-critical inner loops with benchmarked evidence, and FFI boundaries. Document with `// IMPERATIVE: [justification]`

## 9 — Testing benefits of functional code

Pure functions transform the testing pyramid. Because a pure function requires **no setup, no teardown, no mocks, and no database** — just input and expected output — unit testing becomes trivially cheap. Property-based testing becomes practical because any generated input produces a predictable output. The testing strategy for a functional codebase is: **many fast pure-function unit tests (≥80%), few integration tests at the shell boundary (≤15%), minimal end-to-end tests (≤5%).**

### Testing checklist

- [ ] **[CRITICAL]** All pure functions have unit tests — no mock objects, no test doubles, no dependency injection in tests. Pure function tests consist exclusively of input → expected output assertions
- [ ] **[HIGH]** Pure transformation functions have table-driven tests with **≥5 cases** covering happy path, edge cases, boundary values, and error conditions
- [ ] **[HIGH]** Critical domain functions (pricing, trading logic, authentication rules, state transitions) have property-based tests using fast-check (TS), Hypothesis (Python), proptest (Rust), or equivalent
- [ ] **[HIGH]** Property-based tests verify algebraic properties where applicable: **round-tripping** (`decode(encode(x)) === x`), **idempotency** (`f(f(x)) === f(x)`), **invariant preservation** (`isSorted(sort(xs))` always true), **commutativity/associativity** where expected
- [ ] **[HIGH]** Zero mocks are used in functional core unit tests — if a core test requires mocks, the code under test is impure and should be refactored
- [ ] **[MEDIUM]** Integration tests at the imperative shell boundary test real I/O paths — database writes, API calls, message publishing — using Testcontainers or equivalent
- [ ] **[MEDIUM]** Mock count across the entire test suite is tracked as a code health metric — a high mock count signals excessive coupling between business logic and side effects
- [ ] **[MEDIUM]** Test execution time for the pure function test suite is **<2 seconds** for suites of up to 1,000 tests — no I/O overhead should be present
- [ ] **[LOW]** Contract tests verify that adapter implementations conform to port interfaces — Pact, Spring Cloud Contract, or type-level verification

**Target test ratio:**

| Test category | Proportion | Speed | Mocks required |
|---|---|---|---|
| Pure function unit tests | ≥80% | Microseconds each | 0 |
| Shell/adapter integration tests | ≤15% | Seconds each | Real services or Testcontainers |
| End-to-end tests | ≤5% | Seconds–minutes | Full environment |

## 10 — AI agent safety through functional patterns

Functional programming is not merely a style preference for AI-generated code — it is a **safety mechanism**. AI coding agents (Claude Code, Shadow Clone, Copilot) are probabilistic systems: the same prompt can produce different results across runs, and hallucination can introduce subtle bugs that pass superficial review. Functional patterns constrain the damage surface of AI-generated code.

**Pure functions limit blast radius.** An AI-generated pure function cannot accidentally corrupt a database, send an unintended email, or mutate shared state. The worst case is a wrong computation — caught by unit tests, type checks, and property-based testing. An AI-generated impure function, by contrast, can perform any side effect the runtime permits.

**Types constrain hallucination.** When the type system enforces `Result<ParsedOrder, ValidationError>` as the return type, AI cannot return a raw string or silently swallow errors. Exhaustive pattern matching forces AI to handle every variant of a union type. Branded types prevent AI from accidentally passing a `UserId` where a `PostId` is expected.

**Immutability enables safe parallelism.** Multi-agent orchestration systems (Shadow Clone, Agent Teams, Squad) run 3–10 agents concurrently on the same codebase. Git worktree isolation provides filesystem separation, but functional patterns provide logical separation: each agent's pure function modifications are self-contained, composable, and non-destructive. Microsoft's Swarm system validates this with "branch-per-agent isolation" achieving "zero filesystem conflicts by design."

### AI agent compliance checklist

- [ ] **[CRITICAL]** AI-generated functions in the domain layer must be pure — AI agents are not permitted to generate side-effectful code in core/domain modules without human approval
- [ ] **[CRITICAL]** AI-generated code must compile and pass type checks before submission — type errors in AI output are treated as generation failures, not review items
- [ ] **[HIGH]** AI-generated functions must include corresponding unit tests — at minimum 3 table-driven test cases per function demonstrating expected behavior
- [ ] **[HIGH]** AI agents must use existing pure functions as building blocks before writing new logic — composition of verified functions is preferred over novel implementations
- [ ] **[HIGH]** AI-generated discriminated unions and pattern matches must be exhaustive — no wildcard defaults that hide unhandled cases
- [ ] **[MEDIUM]** AI agents must flag any generated code that introduces mutation, side effects, or exception throwing with explicit comments explaining why purity was not achievable
- [ ] **[MEDIUM]** Multi-agent workflows use functional decomposition: independent, pure tasks (data layer, validation, business rules, serialization) are assigned to separate agents
- [ ] **[LOW]** AI agent review protocols include a dedicated @reviewer agent that verifies FP compliance on every completed task

## 11 — Language-specific implementation guide

### 11.1 TypeScript

**Compiler configuration (mandatory):**

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "noPropertyAccessFromIndexSignature": true
  }
}
```

**ESLint FP enforcement (eslint-plugin-functional):**

| Rule | Severity | Purpose |
|---|---|---|
| `functional/no-let` | error | Enforce `const` bindings |
| `functional/immutable-data` | error | Prevent object/array mutation |
| `functional/prefer-readonly-type` | warn | Encourage `Readonly<T>` usage |
| `functional/no-loop-statements` | warn | Encourage map/filter/reduce |
| `functional/no-throw-statements` | error | Enforce Result types |
| `functional/no-return-void` | warn | Encourage return values |

- [ ] **[CRITICAL]** `strict: true` is enabled in `tsconfig.json` — no exceptions
- [ ] **[CRITICAL]** `noUncheckedIndexedAccess: true` — array/object index access returns `T | undefined`
- [ ] **[HIGH]** All domain types use `Readonly<T>`, `ReadonlyArray<T>`, or `as const`
- [ ] **[HIGH]** Discriminated unions use the `never` exhaustiveness pattern or `ts-pattern .exhaustive()`
- [ ] **[HIGH]** Error handling uses neverthrow `Result<T, E>` or Effect-TS `Effect<A, E, R>` — not thrown exceptions
- [ ] **[HIGH]** System boundary parsing uses Zod, Effect Schema, or io-ts — raw `JSON.parse()` results are never passed to domain logic
- [ ] **[MEDIUM]** `eslint-plugin-functional` is installed with at minimum the `recommended` preset
- [ ] **[MEDIUM]** Branded types wrap all domain primitives: `type UserId = string & { __brand: 'UserId' }`

### 11.2 Rust

Rust's ownership model provides implicit purity enforcement that other languages lack. The borrow checker's "aliasing XOR mutation" rule prevents data races at compile time. Default immutability (`let` without `mut`) makes the functional path the path of least resistance.

- [ ] **[CRITICAL]** No `unwrap()` or `expect()` in production code paths — use `?` operator, `match`, `map`, `and_then`, or `unwrap_or_else` with meaningful defaults
- [ ] **[CRITICAL]** `Result<T, E>` is used for all fallible operations — panics are reserved for unrecoverable programming errors
- [ ] **[CRITICAL]** Shared state across threads uses `Arc<Mutex<T>>` or `Arc<RwLock<T>>` — never raw `static mut` or `unsafe` shared mutation
- [ ] **[HIGH]** Minimize `let mut` — the majority of bindings should be immutable `let`. Mutable bindings in core logic require justification
- [ ] **[HIGH]** Iterator chains (`.iter().map().filter().collect()`) are preferred over `for` loops for data transformations — they are zero-cost abstractions
- [ ] **[HIGH]** Custom error types use `thiserror` for libraries (typed enums) and `anyhow` for applications (contextual error chaining)
- [ ] **[HIGH]** Pattern matching is exhaustive — no wildcard `_` catch-all in `match` unless the type is non-exhaustive by design
- [ ] **[HIGH]** Newtype pattern (`struct Meters(f64)`) is used for domain primitives — prevents unit mixing and argument transposition
- [ ] **[MEDIUM]** Clippy is run with `clippy::pedantic` in CI — at minimum, `clippy::unwrap_used` is set to `deny`
- [ ] **[MEDIUM]** `Cow<T>` is used where data may or may not need to be cloned — avoids unnecessary allocation in read-heavy paths
- [ ] **[LOW]** `Send` and `Sync` trait bounds are explicitly documented on public types that cross thread boundaries

### 11.3 Python

Python lacks compile-time purity enforcement, making convention and tooling critical. mypy strict mode, frozen dataclasses, and the dry-python/returns library provide the closest approximation to type-safe functional programming.

- [ ] **[CRITICAL]** mypy `--strict` is enabled in all projects — no untyped definitions, no implicit optional, no unchecked returns
- [ ] **[HIGH]** All domain data classes use `@dataclass(frozen=True)` — instances are immutable after construction
- [ ] **[HIGH]** `typing.Final` annotates all constants and configuration values that should not be reassigned
- [ ] **[HIGH]** Error handling uses dry-python/returns `Result[T, E]` and `Maybe[T]` for domain logic — bare `try/except` is confined to the shell/adapter layer
- [ ] **[HIGH]** System boundary parsing uses Pydantic `BaseModel` with `frozen=True` — loose dicts never enter the domain layer
- [ ] **[MEDIUM]** Generators and `itertools` are used for lazy data processing — list comprehensions producing large intermediate results are replaced with generator expressions
- [ ] **[MEDIUM]** `functools.lru_cache` or `functools.cached_property` is used for memoization of pure functions with repeated calls
- [ ] **[MEDIUM]** `Protocol` classes (structural typing) define ports — adapters implement protocols without inheritance
- [ ] **[LOW]** Python 3.10+ `match`/`case` statements are used for pattern matching with typed patterns
- [ ] **[LOW]** The `@safe` decorator (dry-python/returns) wraps functions to automatically convert exceptions to `Result`

### 11.4 Go

Go's design philosophy emphasizes simplicity over abstraction. Functional patterns in Go are idiomatic when they align with Go's conventions — value semantics, explicit error returns, interfaces as small contracts, and table-driven tests.

- [ ] **[CRITICAL]** Every error return is checked — no ignored `err` values. Use `errcheck` linter to enforce
- [ ] **[CRITICAL]** Errors include context via `fmt.Errorf("operation: %w", err)` — raw error forwarding without wrapping is prohibited in production code
- [ ] **[HIGH]** Value semantics are used by default for structs ≤64 bytes — pointer semantics only when mutation is required or structs are large
- [ ] **[HIGH]** Interfaces are defined by the consumer, not the implementer — accept interfaces, return concrete structs
- [ ] **[HIGH]** The functional options pattern is used for complex constructors — `NewServer(WithTimeout(30*time.Second), WithPort(9090))`
- [ ] **[HIGH]** Table-driven tests are the standard pattern — every pure function has a `tests` slice with named test cases
- [ ] **[MEDIUM]** `context.Context` is the first parameter of all functions that perform I/O or may be cancelled — never stored in structs
- [ ] **[MEDIUM]** Channels and goroutines follow the pattern "share memory by communicating" — no shared mutable state between goroutines without explicit synchronization
- [ ] **[LOW]** Sentinel errors are defined as package-level `var ErrNotFound = errors.New("not found")` with `errors.Is` / `errors.As` for checking

### 11.5 Solidity

Solidity's `pure` and `view` keywords are **compiler-enforced** — the Solidity compiler itself guarantees that `pure` functions cannot read or modify state, and `view` functions cannot modify state. This is the strongest purity enforcement of any mainstream language.

- [ ] **[CRITICAL]** All computation functions that do not access contract state are marked `pure` — the compiler enforces that no state reads or writes occur
- [ ] **[CRITICAL]** All read-only functions that access state but do not modify it are marked `view` — saves gas on external calls (zero transaction cost)
- [ ] **[CRITICAL]** Custom errors (`error InsufficientBalance(uint256 available, uint256 required)`) are used instead of `require()` with string messages — saves **~100–150 gas per revert** and reduces contract bytecode size
- [ ] **[HIGH]** `immutable` is used for values set once in the constructor — stored in bytecode, eliminating SLOAD gas cost (~100 gas saved per read)
- [ ] **[HIGH]** `constant` is used for compile-time constants — expression is inlined, cheapest access pattern
- [ ] **[HIGH]** Events serve as the immutable audit trail — all significant state changes emit typed events with indexed parameters for efficient filtering
- [ ] **[MEDIUM]** Modifier patterns enforce preconditions composably — `onlyOwner`, `validAmount`, `nonReentrant` as reusable guards
- [ ] **[LOW]** Struct transformations in `pure` functions return new structs rather than modifying existing ones in storage

## Appendix C — Architecture reference patterns

### The impureim sandwich

The canonical pattern for structuring any operation that involves side effects:

```
1. READ   (impure)  → Fetch data from database, API, file system, clock
2. DECIDE (pure)    → Apply business rules, validate, transform, compute
3. WRITE  (impure)  → Persist results, send notifications, emit events
```

**Critical constraint:** Once step 3 begins, do not return to step 1. If the operation requires reading additional data based on the decision, restructure as multiple sandwiches composed at the shell level.

### Functional core / imperative shell directory structure

```
src/
├── core/               # PURE — no I/O imports permitted
│   ├── domain/         # Domain types, value objects, entities
│   ├── logic/          # Business rules as pure functions
│   └── errors/         # Typed error definitions
├── shell/              # IMPURE — I/O lives here
│   ├── adapters/       # Repository implementations, API clients
│   ├── handlers/       # HTTP/CLI/message handlers (entry points)
│   └── config/         # Environment reading, initialization
└── composition/        # Wiring — connects core to shell
    └── main.ts         # Composition root, DI container
```

**Enforcement rule:** `core/` modules may not import from `shell/` or `composition/`. This can be enforced via ESLint import restrictions, Rust module visibility, or CI directory-dependency checks.

### CQS at the function level

```
Query (pure):   getActiveUsers(users: User[]) → User[]
Command (void): saveUser(repo: UserRepo, user: User) → void
```

**Never:** `getAndUpdateUser()` — this violates CQS by combining a query and a command. Split into `getUser()` + `updateUser()`.

## Appendix D — Glossary

| Term | Definition |
|---|---|
| **Pure function** | A function that is deterministic (same inputs → same outputs) and has no observable side effects |
| **Referential transparency** | An expression can be replaced by its value without changing program behavior |
| **Side effect** | Any interaction with the external world: I/O, mutation of external state, logging, time reads |
| **Immutability** | Data that cannot be modified after creation — modifications produce new values |
| **Result type** | A discriminated union encoding either a success value or a typed error (also: Either) |
| **Option type** | A discriminated union encoding either a present value or absence (also: Maybe) |
| **Branded type** | A type-level wrapper that distinguishes semantically different primitives at compile time |
| **Newtype** | A zero-cost wrapper type that provides type-level distinction without runtime overhead |
| **Discriminated union** | A type that is one of several tagged variants, each potentially carrying different data (also: ADT, tagged union, sum type) |
| **Exhaustive matching** | Pattern matching that handles every possible variant of a union type, verified by the compiler |
| **Functional core** | The pure business logic layer with no external dependencies or side effects |
| **Imperative shell** | The thin adapter layer that performs I/O and connects the pure core to the external world |
| **Railway-oriented programming** | Composing fallible operations via monadic chaining, where errors bypass subsequent operations |
| **CQS** | Command-Query Separation: functions either return data (query) or perform actions (command), never both |
| **Effect system** | A framework that makes side effects visible and trackable in the type system |
| **Property-based testing** | Testing that generates random inputs and verifies invariant properties hold for all inputs |
| **Idempotency** | An operation that produces the same result when applied multiple times as when applied once |
| **Structural sharing** | An implementation technique where immutable data structures share unchanged subtrees with previous versions |
| **Parse, don't validate** | Transform unstructured input into typed domain objects at boundaries rather than checking validity with boolean functions |
| **Impureim sandwich** | Read (impure) → Calculate (pure) → Write (impure) — the standard operation structure |
